# Repository Instructions

- Treat `.codex/` as persistent working memory for this repository.
- At the start of each task, read `.codex/DEFAULT_INSTRUCTIONS.md` and then read any other `.codex` files that are relevant to the task.
- For planning, architecture, product-definition, or memory-system work, read `.codex/PLAN.md`, `.codex/RESEARCH.md`, `.codex/RESEARCH_INTAKE.md`, `docs/PRD.md`, and `docs/research.md` in addition to the core status files.
- Keep `.codex/STATUS.md` current when the project state changes.
- Record durable architectural or workflow choices in `.codex/DECISIONS.md`.
- Record new limits, guardrails, and external requirements in `.codex/CONSTRAINTS.md`.
- Record hard-won mistakes, surprises, and reusable patterns in `.codex/LESSONS_LEARNED.md`.
- Update `.codex/DESIGN.md` when the intended architecture or implementation plan changes.
- Update `.codex/PLAN.md` when sequencing, priorities, or implementation phases change materially.
- Update `.codex/RESEARCH.md` when external research materially changes the design direction.
- Update `.codex/RESEARCH_INTAKE.md` when new papers, whitepapers, or design references are assessed.
- Update `docs/research.md` when the durable human-readable research synthesis changes materially.
- Update `Changelog.md` when the project has a notable user-visible, workflow, or architecture change worth recording.
- Update `.codex/TASKS.md` when priorities, open work, or next steps change.
- Keep the `.codex` notes concise, factual, and in sync with the actual repository state.
- Do not silently let `.codex` drift; if code and docs disagree, either fix the docs in the same task or note the mismatch in `.codex/STATUS.md`.
